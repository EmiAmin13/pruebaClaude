/undo es la red de seguridad que hace cómodo dejar trabajar al agente: no solo revierte el texto, también los archivos que tocó.

OpenCode trae dos agentes principales. Build es el de por defecto y tiene todas las herramientas habilitadas. Plan es un agente restringido para analizar y proponer.

Conviene precisar algo que suele contarse mal: Plan no es un modo de solo lectura. Funciona con un sistema de permisos que deja las escrituras y los comandos de shell en preguntar, de modo que nada se ejecuta sin que tú lo apruebes.

La regla práctica: empieza en Plan cuando no sabes dónde está el problema, y cambia a Build cuando ya sabes qué archivo tocar. Build y Plan no son los únicos: puedes definir agentes propios, como uno de revisión con acceso de lectura o uno de depuración limitado a bash y lectura.

AGENTS.md es el archivo donde escribes las reglas de tu proyecto para que el agente las lea en cada sesión. Es la diferencia entre repetir «usa signals, no decoradores» cada mañana y no tener que decirlo nunca más.

No es documentación para humanos. Es contexto operativo: estructura del repositorio, convenciones que no se deducen del código, comandos para construir y probar, y los límites que no debe cruzar. Puedes generar un primer borrador con /init y luego podarlo.

Lo que mejor funciona es lo específico y comprobable. «Escribe código limpio» no dice nada; «los tests van junto al archivo, como foo.spec.ts» sí.
