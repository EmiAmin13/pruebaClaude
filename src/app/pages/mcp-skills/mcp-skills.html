El Model Context Protocol es un protocolo abierto que estandariza cómo un agente habla con sistemas externos. Antes de MCP, cada integración era código a medida: uno para GitHub, otro para tu base de datos, otro para Jira. Con MCP, cualquier servicio que hable el protocolo queda disponible para cualquier agente que lo hable.

Un servidor MCP expone herramientas que el agente puede invocar. Los hay de dos tipos: remotos, que viven detrás de una URL, y locales, que el agente arranca como un proceso en tu máquina.

Lo importante: MCP no hace al modelo más listo. Le da acceso a información que no tenía y a acciones que no podía ejecutar.

Si MCP le da manos al agente, una skill le da criterio. Un servidor MCP le permite consultar la documentación de una librería; una skill le enseña qué hacer con ella.

No compiten. Este proyecto usa las dos: Context7 como MCP para consultar documentación real, y una skill de diseño para decidir cómo se ve la interfaz.

Context7 resuelve un problema concreto: el modelo aprendió la documentación de tus librerías en un momento dado, y desde entonces han cambiado. Con Context7, en vez de recordar la API, la consulta.

Es un servidor remoto, así que basta con declarar su URL. La clave se lee de una variable de entorno para no acabar en el repositorio.

Un servidor local se declara igual pero con el comando que lo arranca. Este ejemplo expone las herramientas del CLI de Angular.

Merece la pena escribir en AGENTS.md cuándo quieres que lo use, por ejemplo: «antes de usar una API de una librería, consúltala con Context7». Un servidor disponible que el agente no sabe cuándo invocar no sirve de nada.

Una skill es una carpeta con un archivo SKILL.md: un frontmatter que dice cómo se llama y cuándo usarla, y debajo las instrucciones. Nada más.

La magia está en el description. El agente lee las descripciones de todas las skills disponibles y carga el cuerpo solo de la que hace falta. Así puedes tener veinte skills sin gastar contexto en las diecinueve que hoy no vienen al caso.

Las skills de proyecto viven en el repositorio, junto al código que gobiernan.

Instalar una skill es copiar una carpeta. Puedes escribirla tú, copiarla de otro repositorio o instalarla desde un catálogo como skills.sh.

Dos que valen la pena desde el primer día: frontend-design, que evita que cada interfaz salga con el mismo aspecto de plantilla, y find-skills, que busca e instala las demás.

La pregunta útil antes de instalar una: ¿esto es criterio que quiero que aplique siempre, o una capacidad que necesita? Si es lo segundo, lo que buscas es un servidor MCP.
